# 功能实现标准 V1.0

> 来源：AI 原生对话式记账系统 最终汇总方案 V4.0
> 适用范围：智能体功能定义、工具实现规范、功能验收

---

## 1. 智能体定义标准

系统包含五大智能体，每个智能体有明确的运行位置和职责边界。

| Agent | 运行位置 | 核心职责 | 禁止行为 |
|-------|----------|----------|----------|
| **Master Agent** | 本地意图分类 + 云端复杂推理 | 唯一入口，意图解析，任务分发，结果汇总 | 不得直接操作数据库 |
| **Ledger Agent** | 本地工具为主 | 极速记账、查账、文件导入、OCR、转账、退款 | 不得执行安全扫描 |
| **Analyst Agent** | 本地聚合 + 云端分析 | 统计、趋势检测、异常发现、图表配置生成 | 不得修改账单数据 |
| **Coach Agent** | 云端 + 人格参数 | 预算建议、储蓄挑战、情绪陪伴、成就激励 | 不得访问原始交易数据 |
| **Guardian Agent** | 纯本地 | 实时安全扫描、诈骗预警、无用订阅检测、数据脱敏、AI 行为拦截 | 不得向云端发送任何数据 |

### 1.1 Agent 实现要求

- 每个 Agent 为独立模块，可单独测试、部署、更新
- Agent 必须声明自己的工具依赖列表
- 每个 Agent 的错误必须隔离，不得导致其他 Agent 崩溃
- 所有 Agent 的操作必须可审计（写入 audit_log）

## 3. 核心功能需求

### 3.1 纯对话界面

- 零菜单、零表单设计
- 所有操作通过自然语言完成
- AI 主动服务，不等待用户指令
- 图表以 AI 动态生成的卡片形式嵌入对话流

### 3.2 Multi-Agent 协作

- 用户请求 → Master Agent 意图解析 → 路由到子 Agent → 结果汇总返回
- 多 Agent 可并行执行非冲突任务
- Agent 间仅传递结构化摘要，不传递原始数据

### 3.3 本地工具调用

- 金额计算、分类、安全检测完全在设备端完成
- 工具调用结果对 AI 透明，AI 仅接收工具返回的结构化数据
- 保证精确性（无浮点误差）和隐私

### 3.4 AI 动态可视化

- AI 输出 ECharts 配置 JSON
- 系统 WebView 渲染桑基图、雷达图等高级图表
- 图表配置需校验合法性（无恶意脚本注入）

### 3.5 人教 AI 规则

- 用户通过对话教会 AI 分类商家
- 规则永久生效并支持导出分享
- 规则冲突时按置信度 priority 解决

### 3.6 Skill + 记忆 + 人格

- 可插拔技能系统：Skill 注册、启用、禁用、卸载
- 四层记忆：短期（会话）、长期（用户偏好）、情景（事件）、语义（知识）
- 三维人格：严谨度、幽默度、主动度，用户可调节

### 3.7 纯客户端定时提醒

- 无云推送，依赖系统本地通知
- 后台任务触发周期检查
- 通知权限被拒时降级为应用内提示

## 5. 工具实现规范

### 5.1 每个工具必须包含

```typescript
interface ToolDefinition {
  name: string;           // 唯一工具名 (snake_case)
  description: string;    // 功能描述 (供 AI 理解)
  permissionLevel: 0 | 1 | 2;  // 权限级别
  parameters: ToolParameter[];  // 参数定义
  returns: ToolReturn;    // 返回值定义
  timeout: number;        // 超时时间 (ms)，默认 5000
  retryable: boolean;     // 是否可重试
  idempotent: boolean;    // 是否幂等
}
```

### 5.2 工具错误处理

- 所有工具必须返回统一错误格式：`{ error: string, code: number, retryable: boolean }`
- 工具异常不得抛出未捕获异常，必须 catch 后返回错误对象
- 超时自动终止并返回 `{ error: "timeout", code: 408, retryable: true }`

## 6. 自动分类引擎深度设计

### 6.1 分类架构总览

```
用户输入 (文本/OCR/文件)
       │
       ▼
┌──────────────────────────────────────┐
│        分类流水线 (Classification      │
│               Pipeline)              │
│                                      │
│  ① 预处理 → ② 规则匹配 → ③ ML推理    │
│       │           │           │       │
│       ▼           ▼           ▼       │
│  归一化/分词   确定性匹配   概率分类    │
│  (本地 <1ms)  (本地 <5ms)  (本地/云端) │
│                                      │
│  ④ 后处理 → ⑤ 置信度融合 → ⑥ 输出    │
└──────────────────────────────────────┘
       │
       ▼
  最终分类 + 置信度 + 解释
```

### 6.2 分类策略层次（优先级从高到低）

| 优先级 | 策略 | 引擎 | 延迟 | 置信度 | 适用场景 |
|--------|------|------|------|--------|----------|
| P0 | 确定性规则匹配 | 本地规则引擎 | < 1ms | 1.0 | 用户手动设定的精确规则 |
| P1 | 历史高频模式 | 本地 Trie + LRU 缓存 | < 5ms | ≥ 0.95 | 最近 30 天内 ≥ 3 次的商户 |
| P2 | 商户名模糊匹配 | 本地编辑距离 + 拼音 | < 10ms | 0.7-0.95 | 同商户不同写法（如"KFC"/"肯德基"） |
| P3 | 描述语义分类 | 本地轻量 NLU 模型 | < 50ms | 0.6-0.9 | 无规则匹配时的首猜 |
| P4 | 云端 LLM 推理 | GPT-4o / Claude | < 2s | 0.5-0.85 | 新商户、模糊描述、批量待分类 |

### 6.3 分类规则引擎

#### 6.3.1 规则数据结构

```typescript
interface ClassificationRule {
  id: string;
  pattern: string;               // 匹配模式
  patternType: 'exact' | 'contains' | 'regex' | 'pinyin' | 'levenshtein';
  categoryId: string;
  merchant: string;
  tags: string[];
  confidence: number;            // 0.0 - 1.0
  priority: number;              // 优先级，越高越优先
  scope: 'global' | 'merchant' | 'amount_range';
  amountMin?: number;
  amountMax?: number;
  createdAt: string;
  updatedAt: string;
  hitCount: number;              // 命中次数（用于排序）
  lastHitAt: string;
  source: 'user_taught' | 'system_learned' | 'imported';
}
```

#### 6.3.2 规则匹配算法

```
输入: 账单描述 + 金额
  │
  ▼
Step 1: 精确匹配 (O(1))
  对商户名做 hash → 查精确规则表
  命中 → 直接返回，结束
  
  │ 未命中
  ▼
Step 2: 包含匹配 (O(n))
  遍历 contains 类型规则
  命中 → 按 priority 排序返回最高者
  
  │ 未命中
  ▼
Step 3: 模糊匹配 (O(n))
  Levenshtein 编辑距离 ≤ 2 的商户名
  + 拼音首字母匹配
  命中 → 返回最高 confidence 者
  
  │ 未命中
  ▼
Step 4: 历史推断 (O(log n))
  查询近 90 天相同描述的账单
  命中 → 返回历史上最高频分类
  
  │ 未命中
  ▼
Step 5: ML/NLU 推理 → 返回 Top 3 候选
```

#### 6.3.3 规则冲突解决

| 冲突类型 | 解决策略 |
|----------|----------|
| 同商户多规则 | 取 priority 最高者；相同则取 confidence 最高者 |
| 正则 vs 精确 | 精确匹配优先 |
| 用户规则 vs 系统学习规则 | 用户规则优先（source='user_taught'） |
| 新规则覆盖范围包含旧规则 | 提示用户选择，或保留两者并按 priority 排序 |
| 规则回溯影响 > 50 条历史 | Guardian Agent 拦截，逐条确认 |

### 6.5 批量导入分类工作流

```
CSV/Excel 文件导入 (1000行)
  │
  ▼
Step 1: parse_tabular_file → 列结构识别
  │
  ▼
Step 2: guess_mapping_from_history → 自动列映射
  │
  ▼
Step 3: 逐行分类 (并行 4 线程)
  ├─ 命中规则 → 直接分类 (~80% 行)
  ├─ 历史匹配 → 高置信度 (~10% 行)
  └─ 未匹配 → 标记 "待分类" (~10% 行)
  │
  ▼
Step 4: 待分类行 → 云端 LLM 批量推理 (一次请求，最多 50 行)
  │
  ▼
Step 5: 汇总卡片
  ┌─────────────────────────────────────┐
  │ 📥 导入完成                           │
  │ 总计: 1000 条                        │
  │ ✅ 自动分类: 980 条 (98%)             │
  │ ⚠️ 待确认: 20 条                     │
  │ [一键确认剩余] [逐条审核] [导出报告]    │
  └─────────────────────────────────────┘
```

### 6.6 分类质量闭环

```
用户行为                      系统学习
────────                    ────────
用户手动修改分类     ───→    记录修正对 (描述, 旧分类, 新分类)
用户确认规则建议     ───→    提升规则 confidence (+0.05)
用户拒绝规则建议     ───→    降低规则 confidence (-0.10)
用户频繁使用某分类   ───→    提升该分类在模糊匹配中的权重
用户 30 天不用某规则 ───→    标记为 "休眠"，匹配时降低优先级
```

### 7.2 意图路由矩阵

Master Agent 根据用户输入将请求路由到对应 Agent：

| 意图类别 | 关键词/模式 | 目标 Agent | 示例 |

| **查账** | 查/看/找/搜 + 账单/记录/消费 | Ledger | "查上个月餐饮" |

### 7.4 模糊意图消歧策略

当 Master Agent 无法确定用户意图时：

| 策略 | 示例 | 优先级 |
|------|------|--------|
| **上下文推断** | 刚记完账说"不对" → 修改上一笔 | 1 |
| **主动追问** | "你是想查账单还是设预算？" | 2 |
| **多项候选** | Quick Bar 展示 Top 3 意图供选择 | 3 |
| **安全兜底** | 涉及删除/修改 → 走 Guardian 确认 | 4 |

### 7.5 AI 回复质量标准

| 维度 | 标准 | 反例 |
|------|------|------|
| **简洁** | 非分析类回复 ≤ 3 句话 | 记账后长篇大论 |
| **准确** | 金额、日期、分类 100% 准确 | "大概花了 30 左右" |
| **有用** | 每条回复附带可操作建议 | "你花了不少钱"（无后续） |
| **共情** | 识别用户情绪并适当回应 | 用户抱怨超支，回复"已记录" |
| **安全** | 不明示或暗示原始敏感数据 | 在回复中列出最近 10 笔金额 |

### 7.6 AI 主动服务策略

### 8.2 风险评估算法

```typescript
interface RiskAssessment {
  score: number;           // 0-100，>70 为中风险，>90 为高风险
  level: 'low' | 'medium' | 'high';
  factors: RiskFactor[];
  recommendation: 'auto_approve' | 'confirm' | 'block';
}

interface RiskFactor {
  name: string;
  weight: number;          // 对总分贡献权重
  score: number;           // 该因子得分 0-100
  detail: string;          // 人类可读描述
}

// 风险因子权重
const RISK_WEIGHTS = {
  amountAnomaly:    0.30,   // 金额异常
  frequency:        0.20,   // 频率异常
  impactScope:      0.20,   // 影响范围
  ruleConflict:     0.15,   // 规则冲突
  timeOfDay:        0.05,   // 非正常时段
  newMerchant:      0.10,   // 新商户
};
```

### 8.3 确认卡片交互规范

| 属性 | 要求 |
|------|------|
| 出现方式 | 对话流内插入，非模态弹窗 |
| 超时时间 | 60 秒未响应 → 自动拒绝 |
| 默认焦点 | 始终聚焦在"取消/拒绝"按钮（安全优先） |
| 可跳过 | 低风险 1 级操作可提供"信任此操作，以后跳过"复选框 |
| 批量确认 | ≥ 3 个待确认操作 → 提供"一键全部确认"+"逐条审核" |
| 撤销能力 | 确认后 30 秒内可撤销（软删除标记） |

### 8.5 审批历史与可审计性

- 每次审批决策写入 `approval_log` 表
- 记录：操作内容、风险评估分数、用户决策、决策耗时
- 可导出为审批报告（CSV/JSON）
- 用于优化风险评估模型（用户习惯学习）
