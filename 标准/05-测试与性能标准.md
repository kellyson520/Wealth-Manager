## 1. 性能指标标准

### 1.1 响应时间要求

| 场景 | 实现方式 | Token 消耗 | 响应时间 | 优先级 |
|------|----------|------------|----------|--------|
| 简单记账 | 本地 NLU 直接写库 | 0 | < 100ms | P0 |
| 规则教学 | 云端生成规则，本地存储 | ~200 | < 2s | P1 |
| 月度报表 | 本地聚合，云端生成图表配置 | ~400 | < 2s | P1 |
| 深度分析+建议 | Multi-Agent 仅传递摘要 | ~600 | < 3s | P2 |
| 安全扫描 | 纯本地规则引擎 | 0 | 实时（< 50ms） | P0 |
| 1000行 CSV 导入 | 本地解析+分类+安全检测 | 0 | < 3s | P1 |
| WebDAV 备份 | 加密+上传 | 0 | < 30s（100MB数据） | P2 |
| WebDAV 恢复 | 下载+解密+校验 | 0 | < 60s（100MB数据） | P2 |

### 1.2 Token 消耗目标

| 用户类型 | 月度 Token 消耗上限 | 说明 |
|----------|---------------------|------|
| 轻度用户 | < 10K | 每日记账 ≤ 5 笔 |
| 中度用户 | < 30K | 每日记账 ≤ 15 笔 + 周报 |
| 重度用户 | < 50K | 每日记账 ≤ 30 笔 + 日报 + 分析 |

### 1.3 本地资源限制

| 资源 | 限制 | 说明 |
|------|------|------|
| 数据库大小 | < 500MB（含图片） | 超出提示归档 |
| 内存占用 | < 150MB（运行时） | 后台降为 < 50MB |
| CPU 占用 | 后台 < 5% | 备份任务在后台低优先级执行 |
| 电池消耗 | 后台 < 2% / 小时 | 仅在 WiFi 下执行备份 |

## 2. 测试策略

### 2.1 测试金字塔

```
        ┌──────┐
        │ E2E  │  关键用户旅程
       ┌┴──────┴┐
       │ 集成测试 │  Agent 协作 + 工具链
      ┌┴─────────┴┐
      │  单元测试   │  每个工具独立测试
     ┌┴────────────┴┐
     │  静态分析     │  TypeScript + ESLint
    └───────────────┘
```

### 2.2 各层测试要求

| 层级 | 覆盖率要求 | 工具 | 说明 |
|------|-----------|------|------|
| 静态分析 | 100% 文件 | TypeScript strict mode + ESLint | 类型安全 |
| 单元测试 | ≥ 90% 行覆盖率 | Jest / Vitest | 每个工具至少 3 个用例 |
| 集成测试 | 核心流程全覆盖 | 自定义 test harness | Agent 间通信 + 工具链调用 |
| E2E 测试 | ≥ 10 个关键旅程 | Detox / Appium | 记账→查询→分析→备份 全流程 |

### 2.3 必须覆盖的测试场景

#### 单元测试

- [ ] 每个工具的 happy path（正常输入 → 预期输出）
- [ ] 每个工具的边界条件（空值、极值、超长字符串）
- [ ] 每个工具的错误路径（无效参数、权限不足、超时）
- [ ] 哈希链计算与验证
- [ ] AES-256 加密/解密
- [ ] 规则匹配引擎（正则 + 关键词）

#### 集成测试

- [ ] Master Agent 意图路由（至少 20 种意图）
- [ ] Ledger Agent 完整记账流程（含退款、转账、拆分）
- [ ] Analyst Agent 图表配置生成（至少 6 种图表类型）
- [ ] Guardian Agent 安全拦截（大额、高频、批量、隐私）
- [ ] Multi-Agent 并行协作（记账 + 分析同时进行）
- [ ] WebDAV 备份与恢复全流程

#### E2E 测试

- [ ] 新用户引导 → 首次记账 → 查看报表
- [ ] 批量导入 CSV → 规则匹配 → 自动分类
- [ ] 设定预算 → 超预算预警 → 调整预算
- [ ] 触发安全熔断 → 安全模式 → 恢复
- [ ] 配置 WebDAV → 自动备份 → 换设备恢复

## 3. 安全测试标准

### 3.1 必须执行的攻击测试

| 攻击类型 | 测试方法 | 预期结果 |
|----------|----------|----------|
| Prompt 注入 | 在记账描述中嵌入 "忽略之前指令" | Guardian 拦截或无害化处理 |
| 批量删除攻击 | AI 尝试连续调用 delete_bill × 10 | 第 1 次后触发确认，连续 3 次触发熔断 |
| 隐私提取攻击 | AI 尝试在回复中输出原始数据 | Guardian 自动脱敏 |
| 哈希链篡改 | 手动修改 SQLite 账单记录 | `verify_hash_chain` 检测到断裂 |
| Token 预算绕过 | AI 尝试修改本地 Token 计数器 | 计数器在 AI 不可达的本地沙盒中 |
| WebDAV 中间人 | 模拟自签名证书 | 连接被拒绝（证书绑定） |

### 3.2 模糊测试 (Fuzz Testing)

- 每个接受用户输入的工具进行模糊测试
- 输入：随机字符串、超长字符串、Unicode 特殊字符、SQL 注入片段
- 预期：不崩溃、不泄露数据、不产生错误账单

## 4. 性能测试标准

### 4.1 压力测试

| 场景 | 条件 | 要求 |
|------|------|------|
| 大量账单查询 | 100,000 条账单 | 搜索 < 500ms |

| 测试 | 条件 | 要求 |
|------|------|------|
| 长时间运行 | 连续运行 24 小时 | 内存不泄漏，无崩溃 |

| 网络中断恢复 | 云端调用中切换飞行模式 | 优雅降级，本地功能不受影响 |

## 7. CI/CD 质量门禁

```
提交代码
  ↓
静态分析 (TypeScript + ESLint) ── 失败 → 拒绝合并
  ↓
单元测试 (覆盖率 ≥ 90%) ── 失败 → 拒绝合并
  ↓
集成测试 (核心流程) ── 失败 → 拒绝合并
  ↓
安全扫描 (Guardian 规则集) ── 发现漏洞 → 拒绝合并
  ↓
构建 & 部署到预发布
  ↓
E2E 烟雾测试 ── 失败 → 回滚
  ↓
发布到生产
```
