## 1. 统一错误码体系

### 1.1 错误码编码规则

```
格式: {领域}{子系统}{具体错误}
      1位   2位     3位

领域:
  1xxx - 账单 (Bill)
  2xxx - 分类规则 (Rule)
  3xxx - 资产账户 (Asset)
  4xxx - 安全 (Security)
  5xxx - 备份 (Backup)
  6xxx - 导入导出 (Import/Export)
  7xxx - AI/云端 (AI/Cloud)
  8xxx - 系统 (System)
  9xxx - 通用 (General)
```

## 2. 异常处理标准

### 2.1 处理层级

```
┌────────────────────────────────┐
│ Layer 4: UI 层                 │
│ 展示用户友好的错误提示           │
│ 提供重试/降级/反馈按钮          │
└────────────┬───────────────────┘
             │
┌────────────▼───────────────────┐
│ Layer 3: Agent 层              │
│ 捕获工具异常 → 尝试备用方案      │
│ 记录上下文 → 向上抛标准化错误    │
└────────────┬───────────────────┘
             │
┌────────────▼───────────────────┐
│ Layer 2: 工具层                │
│ try-catch 所有异常              │
│ 转换为统一 ToolResult 格式      │
│ 不得抛出未包装的异常             │
└────────────┬───────────────────┘
             │
┌────────────▼───────────────────┐
│ Layer 1: 数据/引擎层            │
│ 抛出明确的技术异常               │
│ 包含调试信息 (仅开发环境)        │
└────────────────────────────────┘
```

### 2.2 异常包装规范

```typescript
class AppError extends Error {
  code: string;         // 错误码，如 "1002"
  retryable: boolean;
  context: Record<string, unknown>; // 调试上下文 (不含敏感数据)
  userMessage: string;  // 用户友好提示
  
  constructor(opts: {
    code: string;
    message: string;
    retryable?: boolean;
    context?: Record<string, unknown>;
    userMessage?: string;
  }) {
    super(opts.message);
    this.code = opts.code;
    this.retryable = opts.retryable ?? false;
    this.context = opts.context ?? {};
    this.userMessage = opts.userMessage ?? opts.message;
  }
}

// 使用示例
throw new AppError({
  code: '1004',
  message: 'Possible duplicate bill: same amount and merchant within 1 minute',
  retryable: false,
  context: { merchant: '星巴克', amount: 38, lastBillId: 'xxx' },
  userMessage: '刚刚已记过相同账单，要再记一笔吗？',
});
```

### 2.3 禁止的做法

- ❌ `throw new Error('something wrong')` — 必须使用 AppError
- ❌ `try { ... } catch (e) { }` — 禁止吞异常
- ❌ `try { ... } catch (e) { console.log(e) }` — 只打日志不处理
- ❌ 在工具中直接 `throw` — 必须 catch 后返回 ToolResult
- ❌ 在错误消息中包含敏感数据 (金额、商户名原始值)

## 3. 日志标准

### 3.1 日志级别

| 级别 | 含义 | 生产环境 | 示例 |
|------|------|----------|------|
| **DEBUG** | 详细调试信息 | ❌ 关闭 | 函数入参、中间计算结果 |
| **INFO** | 关键业务流程 | ✅ 开启 | 记账成功、备份完成、AI 调用 |
| **WARN** | 潜在问题 | ✅ 开启 | Token 80%、规则冲突、网络重试 |
| **ERROR** | 操作失败 | ✅ 开启 | 工具失败、AI 超时、备份失败 |
| **FATAL** | 崩溃级错误 | ✅ 开启 | 数据库损坏、哈希链断裂 |

### 3.2 日志格式

```typescript
interface LogEntry {
  timestamp: string;      // ISO 8601
  level: LogLevel;
  tag: string;            // 模块名: bills/security/ai/backup
  message: string;
  context?: {
    traceId?: string;
    agentId?: string;
    toolName?: string;
    errorCode?: string;
    durationMs?: number;
  };
  // 严禁记录: amount, merchant, password, token, key
}
```

### 3.3 日志存储

| 环境 | 输出 | 保留 |
|------|------|------|
| 开发 | Console + 文件 | 7 天 |
| 预发布 | Console + 文件 + Sentry | 7 天 |
| 生产 | 文件 + Sentry (ERROR/FATAL) | 本地 7 天 / Sentry 90 天 |

### 3.4 必须记录的日志

| 事件 | 级别 | 内容 |
|------|------|------|
| 用户记账 | INFO | traceId, durationMs |
| 工具调用 | INFO | toolName, result status, durationMs |
| 云端 AI 调用 | INFO | model, tokenUsed, durationMs |
| 安全拦截 | WARN | rule triggered, riskScore |
| 熔断触发 | WARN | trigger, context |
| 备份完成/失败 | INFO/ERROR | snapshotId, fileSize, durationMs |
| 数据库迁移 | INFO | fromVersion, toVersion, durationMs |
| App 崩溃 | FATAL | stack trace (去敏) |

## 4. 崩溃恢复

### 4.1 数据库恢复

```
App 启动
  │
  ▼
检测数据库完整性 (PRAGMA integrity_check)
  │
  ├─ 通过 → 正常启动
  │
  └─ 失败
      │
      ▼
  尝试从最近备份恢复
      │
      ├─ 有备份 → 恢复 → 重放 WAL 日志 → 验证哈希链
      │     ├─ 成功 → 通知用户 "数据已自动修复"
      │     └─ 失败 → 提示用户 "数据损坏，请从云端备份恢复"
      │
      └─ 无备份 → 提示用户 "数据库损坏，需要重新开始或从备份恢复"
```

### 4.2 事务完整性

```typescript
// 所有写操作必须在事务中执行
async function executeWithIntegrity<T>(
  operation: () => Promise<T>,
  rollback?: () => Promise<void>
): Promise<T> {
  const beforeHash = await getLatestHashChain();
  
  try {
    await db.exec('BEGIN');
    const result = await operation();
    await db.exec('COMMIT');
    return result;
  } catch (error) {
    await db.exec('ROLLBACK');
    await rollback?.();
    
    // 验证回滚后数据完整性
    const afterHash = await getLatestHashChain();
    if (beforeHash !== afterHash) {
      // 严重：回滚未恢复状态
      await triggerEmergencyMode('hash_chain_mismatch_after_rollback');
    }
    
    throw error;
  }
}
```

### 4.3 未处理异常兜底

```
全局未捕获异常
  │
  ▼
Global Error Handler
  ├─ 记录 FATAL 日志 + 堆栈
  ├─ 尝试保存当前会话状态
  ├─ 确保数据库连接正常关闭
  └─ 展示友好的崩溃提示
      ┌─────────────────────────────┐
      │ 😔 出了点问题                 │
      │ App 遇到了意外错误             │
      │ 您的数据是安全的               │
      │ [重新启动] [发送反馈]         │
      └─────────────────────────────┘
```
